# Agent

Agent 可以理解为“带目标、会拆任务、能使用工具并持续校验结果的 AI 工作流”。在文档项目里，它不只是生成几篇文章，而是要完成信息架构、内容拆分、交互辅助、导航配置和构建验证。

这篇文档沉淀一套适用于 VitePress 技术文档站的 Agent 工作规范。它可以用于 C++、Java、Python、数据库、工具链等主题，其中 C++ 这类知识点密集的主题要特别重视目录细分和可视化解释。

## 适用场景

- 为已有文档站建设或重构一个技术主题。
- 把粗略目录拆成细颗粒度学习路径。
- 补齐示例代码、常见错误、小练习和阶段总结。
- 用 Vue 3、TypeScript、HTML5、CSS3 制作轻量交互组件。
- 更新 VitePress nav、sidebar、首页入口和内部链接。
- 完成后执行构建检查，保证站点可访问。

不适合的场景：

- 只需要一句解释的问题。
- 没有扫描项目结构就批量创建文件。
- 把一次性执行提示词原样塞进公开文档。
- 用长篇复述代替真正的目录设计和内容建设。

## 核心原则

### 1. 先设计信息架构

不要先写正文再补目录。技术文档应先确认：

- 主题属于哪个顶层栏目。
- 目录是否复用现有路径。
- 页面粒度是否足够细。
- sidebar 是否能支撑长期扩展。
- 首页大纲是否能帮助读者选择学习路径。

如果项目已有 `docs/backend/cpp/`，就不要再创建 `docs/cpp/`。Agent 必须跟随项目已有结构，而不是机械执行模板路径。

### 2. 一个页面只讲一组紧密相关的问题

如果一个页面同时塞入数组、二维数组、字符串、字符数组、指针、栈堆、类和算法，它就已经过宽，需要拆分。

拆分判断可以用这个规则：

| 情况 | 处理方式 |
| --- | --- |
| 主题有独立概念、语法、常见错误和练习 | 拆成独立页面 |
| 只是几个相关小点 | 放在同一页面 |
| 内容偏复杂但不适合作为导航页 | 放进 `::: details` 折叠块 |
| 只是承上启下 | 做成导读页，不重复正文 |

### 3. 复杂内容要折叠或可视化

目录大纲、速查表、扩展说明、错误分析适合使用 VitePress 的 `::: details`。抽象过程、状态变化和空间关系适合做交互组件。

不要用大段文字解释读者很难想象的过程，例如：

- 编译流程
- 数组下标访问
- 栈帧变化
- 堆内存申请和释放
- 对象创建、成员状态变化
- `vector` 扩容
- 排序过程
- 继承和多态调用

这些内容优先使用 Vue 组件、流程图或小动画辅助理解。

## 可复用配置

每次启动文档建设前，先明确配置。

```yaml
技术主题: C++
文档目录: docs/backend/cpp/
组件前缀: Cpp
目标读者: 零基础或有少量编程经验的学习者
文档风格: 浅显、循序渐进、例子丰富、避免大段理论
项目类型: VitePress 技术文档
文档语言: 中文
组件技术: Vue 3 + TypeScript + HTML5 + CSS3
是否需要练习题: 是
是否需要代码示例: 是
是否需要折叠块: 是
是否需要构建检查: 是
```

切换主题时，只改主题、目录和组件前缀。

```yaml
技术主题: Python
文档目录: docs/backend/python/
组件前缀: Python
```

## 推荐执行流程

### 1. 扫描项目

先检查：

```text
package.json
docs/
docs/.vitepress/
docs/.vitepress/config.ts
docs/.vitepress/theme/
docs/.vitepress/theme/components/
```

同时检查现有主题目录、已有文章、导航和侧边栏。禁止在未确认结构前批量创建新目录。

### 2. 识别问题

对现有文档做一次结构诊断：

- 是否存在只有标题没有正文的占位页。
- 是否存在一个页面承载太多知识点。
- 是否存在内容完全多余、只是在复述执行提示的文章。
- 是否缺少读者必学但目录没有覆盖的基础内容。
- 是否缺少可视化解释的抽象章节。
- sidebar 是否只是平铺列表，难以扩展。

### 3. 设计细分目录

以 C++ 为例，推荐按下面粒度拆分。

```text
docs/backend/cpp/
  index.md
  getting-started.md
  development-environment.md
  input-output.md
  syntax-data-expressions.md
  type-conversion.md
  control-flow.md
  functions-modular.md
  compound-types-memory.md
  arrays.md
  multidimensional-arrays.md
  strings.md
  c-strings.md
  pointers-references-resources.md
  stack-heap-memory.md
  classes-objects.md
  inheritance-polymorphism.md
  templates-generic.md
  stl-templates.md
  common-algorithms.md
  io-files-exceptions.md
  modern-cpp.md
  concurrency.md
  algorithms-data-structures.md
  engineering-practice.md
```

sidebar 建议按学习阶段分组，而不是一层平铺：

```text
入门准备
语法基础
数据组织
内存与资源
面向对象
泛型、STL 与算法
进阶与工程
```

复杂分类可以设置 `collapsed: true`，避免目录压迫感过强。

### 4. 拆分 worker

可以模拟多个 worker，但每个 worker 必须有清晰边界。

| Worker | 职责 | 输出 |
| --- | --- | --- |
| 结构 worker | 目录、sidebar、首页大纲 | `index.md`、`config.ts` |
| 基础 worker | 输入输出、语法、类型转换、控制流 | 基础语法页面 |
| 数据 worker | 数组、二维数组、字符串、字符数组 | 数据组织页面 |
| 内存 worker | 指针、引用、栈、堆、资源管理 | 内存页面和动画 |
| OOP worker | 类、对象、继承、多态 | 面向对象页面和动画 |
| STL worker | 模板、容器、常用算法 | 标准库页面和动画 |
| 工程 worker | 文件、异常、并发、工程实践 | 进阶实践页面 |
| 校对 worker | 链接、重复内容、构建检查 | 修复清单 |

真实执行不一定要真的并发，但要保持文件所有权清楚，避免互相覆盖。

## 页面写作规范

每篇正文建议使用下面结构。

```md
# 标题

## 这一节你会学到什么

- 知识点 1
- 知识点 2
- 知识点 3

## 它是什么？

先用普通语言解释概念。

## 为什么需要它？

结合实际场景说明它解决什么问题。

## 基础用法

给出短小、可运行的代码。

## 常见错误

解释错误原因和修复方式。

## 小练习

给出从简单到稍难的练习。

## 本节小结

用 3 到 5 条要点收束。
```

代码示例要求：

- 标注语言，例如 `cpp`。
- 尽量短小，可以单独运行。
- 不提前大量引入后续章节才讲的语法。
- 每个代码块后要解释关键点。
- 不把所有答案直接堆出来，练习优先给参考思路。

## 折叠块使用规范

适合放进 `::: details` 的内容：

- 大纲中的二级清单。
- 速查表。
- 进阶解释。
- 常见错误深挖。
- 对初学者暂时不是主线但有价值的补充。

示例：

```md
::: details 常用 string 成员函数
| 函数 | 作用 |
| --- | --- |
| `size()` | 获取长度 |
| `find()` | 查找子串 |
| `substr()` | 截取子串 |
:::
```

不要把正文主线全部折叠起来。折叠块是辅助，不是隐藏主要内容。

## 交互组件规范

组件放置位置：

```text
docs/.vitepress/theme/components/
```

命名方式：

```text
CppCompileFlow.vue
CppArrayIndexDemo.vue
CppStackHeapDemo.vue
CppClassObjectDemo.vue
CppVectorGrowthDemo.vue
CppAlgorithmSortDemo.vue
```

组件要求：

- 使用 Vue 3 `<script setup lang="ts">`。
- 使用 TypeScript 表达状态。
- 使用 HTML5 结构表达语义。
- 使用 CSS3 动画或过渡表达变化。
- 使用 VitePress 主题变量：`--vp-c-bg-soft`、`--vp-c-border`、`--vp-c-text-1`、`--vp-c-brand-1`。
- 样式必须 `scoped`。
- 不引入大型 UI 框架。
- 一个组件只解释一个概念。

决策表：

| 知识点 | 推荐表现 |
| --- | --- |
| 编译与链接 | 步骤切换组件或流程图 |
| 数组下标 | 可点击格子 |
| 二维数组 | 行列高亮 |
| 指针与地址 | 地址卡片和解引用提示 |
| 栈与堆 | 栈帧压入/弹出动画 |
| 类与对象 | 对象字段和成员函数状态变化 |
| vector 扩容 | size/capacity 可视化 |
| 排序算法 | 柱状条移动或步骤切换 |
| 多态 | 同一接口调用不同实现 |

## 内容质量标准

合格输出应满足：

- 目录颗粒度足够细，不把复杂主题混成一篇。
- 每篇文章都能独立解决一个明确问题。
- 导读页只做导航，不复述所有正文。
- 首页大纲可展开收缩，读者能快速定位学习阶段。
- 复杂解释有折叠块、图解或交互组件。
- 示例代码完整、短小、可运行。
- 章节之间难度递进，不突然跳到高级语法。
- 不出现只有标题没有正文的页面。
- 不出现和主题无关的 Agent 执行提示复述。
- 不破坏现有导航、主题、首页和构建。

## 质量检查清单

完成后逐项检查：

```text
Markdown 文件是否存在且有正文
首页大纲是否覆盖新增页面
sidebar 是否分组清晰
内部链接是否正确
代码块语言标识是否正确
组件是否已注册
组件是否适配深色 / 浅色模式
复杂内容是否合理使用 details
是否删除或迁移了多余复述内容
构建命令是否通过
```

根据项目脚本执行：

```bash
npm run docs:build
```

如果构建失败，必须先阅读错误信息，定位文件，修复后重新构建。

## 反模式

这些做法应避免：

- 只根据标题生成一批浅文章。
- 把“数组、二维数组、字符串、字符数组、指针、内存、类”放在一个大杂烩页面。
- 在公开文档中复述“你是某某 Agent，请立即开始执行”这类一次性提示。
- sidebar 一层平铺几十个条目，没有学习阶段。
- 为了炫技做复杂动画，却没有帮助读者理解概念。
- 组件样式污染全局，或在深色模式下不可读。
- 构建失败仍然声称完成。
