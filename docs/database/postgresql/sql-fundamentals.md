# SQL 基础

PostgreSQL 的 SQL 方言高度遵循 SQL 标准，同时提供了大量扩展语法。理解 PostgreSQL 特有的语法细节和数据类型，是从其他数据库迁移过来的第一步。

## 标识符规则

PostgreSQL 对标识符的处理与 MySQL 有显著差异：

```sql
-- 不加引号的标识符：自动转为小写
CREATE TABLE MyTable (ID INT);    -- 实际创建的表名是 mytable
SELECT * FROM MyTable;            -- 查询的也是 mytable

-- 加双引号的标识符：保留大小写
CREATE TABLE "MyTable" ("ID" INT);
SELECT * FROM "MyTable";          -- 必须用双引号
SELECT * FROM MyTable;            -- 报错：mytable 不存在

-- 关键字可以用作标识名（需双引号）
CREATE TABLE "user" ("group" INT);
```

::: warning 大小写陷阱
从 MySQL 迁移到 PostgreSQL 时最常见的问题：MySQL 的表名大小写取决于文件系统配置（`lower_case_table_names`），而 PostgreSQL 始终将未引用的标识符转为小写。建议全部使用小写+下划线命名，避免双引号。
:::

## 注释风格

```sql
-- 单行注释（SQL 标准）
/* 多行注释 */

-- PostgreSQL 特有的对象注释
COMMENT ON TABLE orders IS '订单主表';
COMMENT ON COLUMN orders.status IS '订单状态: pending/confirmed/shipped/completed';
COMMENT ON INDEX idx_orders_user_id IS '用户ID索引，用于按用户查询订单';
```

## 数据类型

### 整型与自增

```sql
-- PostgreSQL 没有 TINYINT/SMALLINT/MEDIUMINT 的命名，但有等价类型
-- SMALLINT  = 2 字节（-32768 ~ 32767）
-- INTEGER   = 4 字节
-- BIGINT    = 8 字节

-- SERIAL 系列（自动创建 sequence）
CREATE TABLE items (
    id SERIAL PRIMARY KEY,           -- 等价于 INTEGER + DEFAULT nextval(...)
    name TEXT NOT NULL
);
-- SERIAL 实际上做了：
--   CREATE SEQUENCE items_id_seq;
--   ALTER TABLE items ALTER COLUMN id SET DEFAULT nextval('items_id_seq');

-- BIGSERIAL / SMALLSERIAL 分别对应 BIGINT / SMALLINT

-- PostgreSQL 10+ 推荐使用 GENERATED（SQL 标准语法）
CREATE TABLE items_v2 (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name TEXT NOT NULL
);
-- GENERATED ALWAYS AS IDENTITY 更严格：不允许手动插入 id 值
-- 而 SERIAL 允许 INSERT INTO items(id) VALUES (999)
```

::: tip SERIAL vs IDENTITY
`GENERATED ALWAYS AS IDENTITY` 是 SQL 标准语法，推荐在新项目中使用。如果确实需要手动指定 ID，用 `GENERATED BY DEFAULT AS IDENTITY`，或使用 `OVERRIDING SYSTEM VALUE`。
:::

### UUID 类型

```sql
-- 启用 uuid-ossp 扩展
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 或使用内置的 gen_random_uuid()（PostgreSQL 13+）
CREATE TABLE sessions (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id INTEGER NOT NULL,
    token TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- UUID 的存储：16 字节（比 CHAR(36) 节省空间）
-- 查询时可以直接比较
SELECT * FROM sessions WHERE id = 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11';
```

::: tip UUID 索引性能
UUID 作为主键的 B-Tree 索引，因为值是随机的（非顺序），写入时会导致随机 I/O。对于高写入吞吐的场景，可以考虑使用 `uuid_generate_v7()`（基于时间戳排序的 UUID）或使用 BIGSERIAL 作为内部主键，UUID 作为业务标识。
:::

### 数组类型

PostgreSQL 原生支持数组，这是区别于其他关系数据库的重要特性：

```sql
-- 声明数组列
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    title TEXT NOT NULL,
    tags TEXT[] DEFAULT '{}',              -- TEXT 数组
    scores INTEGER[],                      -- INTEGER 数组
    dimensions DOUBLE PRECISION[3]         -- 固定长度数组
);

-- 插入数据
INSERT INTO articles (title, tags, scores) VALUES
  ('PostgreSQL 入门', ARRAY['database', 'pg', 'tutorial'], ARRAY[90, 85, 92]),
  ('SQL 优化', '{database, optimization}', '{88, 95}');

-- 数组查询
-- @> 包含运算符
SELECT * FROM articles WHERE tags @> ARRAY['database'];

-- && 重叠运算符
SELECT * FROM articles WHERE tags && ARRAY['pg', 'mysql'];

-- ANY 查询
SELECT * FROM articles WHERE 'pg' = ANY(tags);

-- 数组索引（GIN 索引）
CREATE INDEX idx_articles_tags ON articles USING GIN (tags);

-- 数组展开
SELECT id, unnest(tags) AS tag FROM articles;

-- 数组长度与元素访问
SELECT title, array_length(tags, 1) AS tag_count, tags[1] AS first_tag
FROM articles;
```

### RANGE 类型

Range 类型表示一个值的范围，PostgreSQL 内建了多种 range 类型：

```sql
-- 内建 range 类型：int4range, int8range, numrange, tsrange, tstzrange, daterange

-- 创建使用 range 的表
CREATE TABLE reservations (
    id SERIAL PRIMARY KEY,
    room_id INTEGER NOT NULL,
    during TSRANGE NOT NULL,              -- 时间范围
    EXCLUDE USING GIST (room_id WITH =, during WITH &&)  -- 排除约束
);

-- 插入范围值
INSERT INTO reservations (room_id, during) VALUES
  (1, '[2024-01-15 09:00, 2024-01-15 12:00)'),
  (1, '[2024-01-15 14:00, 2024-01-15 17:00]');

-- 范围查询
-- @> 包含某个值
SELECT * FROM reservations WHERE during @> '2024-01-15 10:00'::TIMESTAMP;

-- && 范围重叠
SELECT * FROM reservations WHERE during && '[2024-01-15 11:00, 2024-01-15 15:00)';

-- 上下界操作
SELECT lower(during), upper(during), isempty(during) FROM reservations;
```

::: tip EXCLUDE 约束
`EXCLUDE USING GIST (room_id WITH =, during WITH &&)` 确保同一房间的时间范围不重叠。这比在应用层做冲突检测更可靠，是 PostgreSQL 独有的约束能力。
:::

### 时间类型

```sql
-- TIMESTAMPTZ（推荐） vs TIMESTAMP
-- TIMESTAMPTZ 存储时为 UTC，检索时按 session timezone 转换
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    name TEXT,
    occurred_at TIMESTAMPTZ DEFAULT NOW()
);

-- 设置 session 时区
SET TIME ZONE 'Asia/Shanghai';
SELECT NOW();                  -- 返回带时区的时间
SELECT NOW()::DATE;            -- 日期转换

-- 时间运算
SELECT NOW() + INTERVAL '7 days';
SELECT NOW() - INTERVAL '3 hours';
SELECT EXTRACT(EPOCH FROM NOW() - created_at) AS seconds_ago FROM events;

-- date_trunc 按粒度截断
SELECT date_trunc('month', NOW());
SELECT date_trunc('hour', TIMESTAMPTZ '2024-01-15 10:30:45+08');
```

### JSONB vs JSON

```sql
-- JSON：存储原始文本，每次查询需要解析
-- JSONB：存储为二进制格式，支持索引，去重键
CREATE TABLE api_responses (
    id SERIAL PRIMARY KEY,
    payload JSONB NOT NULL
);
```

## 布尔类型

```sql
-- PostgreSQL 有真正的布尔类型
CREATE TABLE settings (
    user_id INTEGER PRIMARY KEY,
    dark_mode BOOLEAN DEFAULT FALSE,
    notifications BOOLEAN DEFAULT TRUE
);

-- 布尔值字面量：TRUE / FALSE / NULL
INSERT INTO settings VALUES (1, TRUE, FALSE);
INSERT INTO settings VALUES (2, 'yes', 'off');    -- 也可以用 yes/no/on/off

-- WHERE 中的布尔用法
SELECT * FROM settings WHERE dark_mode;           -- 等价于 WHERE dark_mode = TRUE
SELECT * FROM settings WHERE NOT notifications;
```

## 模式（Schema）

PostgreSQL 的 Schema 是命名空间，不是 MySQL 中的"数据库"概念：

```sql
-- PostgreSQL 的层级关系：Database → Schema → Table
-- MySQL 的层级关系：Database → Table（无 Schema 概念）

-- 创建 schema
CREATE SCHEMA analytics;
CREATE SCHEMA IF NOT EXISTS auth;

-- 在指定 schema 中创建表
CREATE TABLE analytics.page_views (
    id BIGSERIAL PRIMARY KEY,
    url TEXT NOT NULL,
    viewed_at TIMESTAMPTZ DEFAULT NOW()
);

-- 搜索路径
SHOW search_path;                  -- 默认: "$user", public
SET search_path TO analytics, public;

-- 不指定 schema 时，按 search_path 查找
SELECT * FROM page_views;          -- 如果 search_path 包含 analytics

-- 跨 schema 查询
SELECT * FROM auth.users u
JOIN analytics.page_views pv ON pv.user_id = u.id;
```

## RETURNING 子句

PostgreSQL 的 RETURNING 可以在 INSERT/UPDATE/DELETE 时直接返回受影响的行，避免额外的 SELECT：

```sql
-- INSERT 返回自动生成的值
INSERT INTO users (name, email) VALUES ('Alice', 'alice@example.com')
RETURNING id, name, created_at;
-- 返回: 1 | Alice | 2024-01-15 10:30:00+08

-- UPDATE 返回修改后的值
UPDATE users SET email = 'new@example.com' WHERE id = 1
RETURNING id, email AS new_email;

-- DELETE 返回被删除的行
DELETE FROM sessions WHERE expired_at < NOW()
RETURNING id, user_id;

-- 实际场景：原子递增计数器
UPDATE counters SET value = value + 1 WHERE name = 'order_seq'
RETURNING value AS next_order_id;
```

## UPSERT (ON CONFLICT)

```sql
-- 插入或更新（类似 MySQL 的 INSERT ... ON DUPLICATE KEY UPDATE）
CREATE TABLE user_settings (
    user_id INTEGER PRIMARY KEY,
    theme TEXT DEFAULT 'light',
    lang TEXT DEFAULT 'zh-CN',
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 如果 user_id 冲突，更新 theme 和 lang
INSERT INTO user_settings (user_id, theme, lang)
VALUES (1, 'dark', 'en')
ON CONFLICT (user_id)
DO UPDATE SET
    theme = EXCLUDED.theme,
    lang = EXCLUDED.lang,
    updated_at = NOW();

-- EXCLUDED 引用的是 INSERT 语句中尝试插入的值

-- 冲突时什么都不做
INSERT INTO user_settings (user_id) VALUES (1)
ON CONFLICT (user_id) DO NOTHING;
```

::: tip ON CONFLICT 的目标
`ON CONFLICT` 必须指定一个唯一约束或唯一索引作为目标。如果表上有多个唯一约束，需要显式指定：`ON CONFLICT (user_id) DO UPDATE ...` 或 `ON CONSTRAINT constraint_name DO UPDATE ...`。
:::
